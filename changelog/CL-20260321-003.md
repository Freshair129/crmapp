## Design Decisions สำคัญ

### Role Hierarchy ใหม่ (8 roles)
```
L0  DEVELOPER   → Full everything + System config
L1  MANAGER     → Full business + approve Ads/PR
L2  ADMIN       → Operations (no marketing, no system)
L2.5 MARKETING  → Marketing domain full + inbox view  ← ใหม่
L2.5 HEAD_CHEF  → Kitchen domain full + enrollment    ← ใหม่
L3  EMPLOYEE    → Day-to-day + own data
L4  AGENT       → Own inbox + own customers + POS create
L5  GUEST       → Read-only dashboard
```

### Permission Matrix
- แทน hardcoded `['ADMIN','MANAGER'].includes(role)` ทั่ว codebase
- ใช้ `can(role, domain, action)` จาก `src/lib/permissionMatrix.js`
- 7 domains: business, sales, inbox, marketing, kitchen, catalog, system

### Ads Optimize
- Meta `ads_management` permission มีอยู่แล้ว → เปิด write operations
- Safe actions (pause/resume/duplicate): MARKETING ทำได้เลย
- Budget/Bid: MARKETING ทำได้ + audit log
- Lifetime Budget: ต้อง MANAGER approve (approval workflow)
- ทุก write action → บันทึกใน AuditLog

## Breaking Changes (จะเกิดตอน implement 29a)

1. **DB role values** → ต้อง normalize เป็น UPPERCASE ทั้งหมด
2. **Force re-login** → เปลี่ยน `NEXTAUTH_SECRET` เพื่อ expire sessions เดิม
3. **Prisma schema** → เพิ่ม `AdsOptimizeRequest` model
